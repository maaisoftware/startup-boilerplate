# seo-audit

SEO enforcement in this template is not optional. Every route answers six questions; this skill walks through them.

## Scope

Run against a single route or against every route listed in `/sitemap.xml`. For a single-route audit, the user provides the path (e.g. `/blog/hello-world`).

## The six checks

### 1. Metadata export

Open the route file (e.g. `apps/web/src/app/blog/[slug]/page.tsx`). It must export one of:

- `export const metadata: Metadata = { ... }` for static pages.
- `export async function generateMetadata({ params }): Promise<Metadata> { ... }` for dynamic pages.

The Metadata must include at minimum:

- `title` (short, human, doesn't need to include the site name — the root layout's template handles that)
- `description` (150–160 chars)
- `openGraph` with `type`, `title`, `description`
- `twitter` with `card: "summary_large_image"`

Missing metadata → FAIL. After PR #17, the `no-page-without-metadata` ESLint rule blocks CI.

### 2. PageShell + JSON-LD

The rendered page must use `<PageShell>` from `@startup-boilerplate/ui`. PageShell requires:

- `title`, `description` as TypeScript-enforced props (must match the metadata export).
- `structuredData` as a Schema.org JSON-LD payload. Use the builders:
  - `articleSchema({ headline, description, url, author?, datePublished?, dateModified? })` for posts
  - `organizationSchema({ name, url, logo?, sameAs? })` for the root
  - `breadcrumbSchema([{ name, url }])` for listings

Fetch the page in a browser or via `curl`. Verify `<script type="application/ld+json">` is present. Paste the JSON into Google's [Rich Results Test](https://search.google.com/test/rich-results) to validate.

### 3. Canonical URL

The root layout sets `metadataBase` to `NEXT_PUBLIC_APP_URL`. Next.js derives `canonical` from this unless the page overrides it. For detail pages, explicitly set:

```ts
alternates: {
  canonical: `/blog/${slug}`;
}
```

Verify by inspecting `<link rel="canonical">` on the rendered HTML.

### 4. OG + Twitter tags

With `metadataBase` set and the `openGraph`/`twitter` fields populated in `generateMetadata`, Next.js emits:

- `<meta property="og:title">`
- `<meta property="og:description">`
- `<meta property="og:url">`
- `<meta property="og:type">`
- `<meta property="og:image">` (if `images: [...]` is set)
- `<meta name="twitter:card" content="summary_large_image">`
- `<meta name="twitter:title">`, etc.

Fail if any of the first four are missing. Image is strongly recommended but not blocking.

### 5. Sitemap coverage

Open `/sitemap.xml`. It's generated by `apps/web/src/app/sitemap.ts` which iterates the CMS. Verify the new route appears. If it's a dynamic route, confirm the CMS method that backs the sitemap lists the new resource — e.g. `listPublishedPosts` for `/blog/*`.

### 6. Robots + llms.txt alignment

- `/robots.txt` must allow the path (or explicitly disallow it if it's admin-only).
- `/llms.txt` should list the path under primary paths for public content, or under "Disallowed" for staff-only.

Both are generated. Inspect `apps/web/src/app/robots.ts` and `apps/web/src/app/llms.txt/route.ts` — they use `NEXT_PUBLIC_APP_URL` as origin.

## Tooling

- Google Rich Results Test: https://search.google.com/test/rich-results
- Meta Sharing Debugger: https://developers.facebook.com/tools/debug/
- Twitter Card Validator: https://cards-dev.twitter.com/validator
- `curl -s "<url>" | grep -E 'og:|twitter:|application/ld+json|canonical'` — quick local check.

## Output format

```
Route: /blog/hello-world
--------------------------------
✓ metadata export: generateMetadata present
✓ PageShell: rendered with articleSchema JSON-LD
✗ canonical: missing — add `alternates.canonical` in generateMetadata
✓ OG/Twitter: all four required tags present
✓ sitemap: listed
✓ robots: allowed
✓ llms.txt: listed under primary paths

1 failure. Fix: set `alternates: { canonical: '/blog/' + slug }` in generateMetadata.
```

## Playwright coverage

The smoke suite at `apps/web/tests/e2e/smoke.spec.ts` has a subset of these checks — OG title, JSON-LD presence, sitemap content. Extend that suite when you add a new route type so regressions are caught automatically.
